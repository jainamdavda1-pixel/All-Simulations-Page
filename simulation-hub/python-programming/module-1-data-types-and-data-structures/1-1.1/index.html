THEORYDynamic Typing & Implicit Coercion

Core Concepts:

In Python, types are bound to values (objects), not variables. Variables are just name references pointing to objects in memory.

  • Dynamic vs. Static Typing: Static typed languages (like Java/C++) enforce type definitions on variables during compilation. Python is dynamically typed; it determines object types at runtime by inspection.
  • Implicit Coercion & Subclasses: Python values can dynamically coerce in mixed expressions. For example, True + 42 evaluates to 43 because bool is implemented as a direct subclass of int (where True represents 1, and False represents 0). Likewise, 3 + 4.0 evaluates to the float 7.0 via implicit widening.
  • Explicit Conversion Barriers: Some operations are syntactically impossible to evaluate dynamically and raise errors. Calling int("3.14") throws a ValueError because string literal decimal points cannot be directly converted to integers without first being parsed as a float.

THEORYIdentifier Constraints & PEP 8 Style Conventions

Core Concepts:

Identifiers are the names assigned to variables, classes, functions, etc. Python enforces rigid rules on how identifiers must be structured:

  • Lexical Rules: An identifier must begin with a letter (A-Z, a-z) or an underscore (_), followed by letters, digits (0-9), or underscores. Symbols like hyphens (-) and periods are lexical errors. Starting with a digit (e.g., 2nd_value) is invalid because the parser expects it to be a numeric literal.
  • Parsing Keywords: Python reserves 35 keywords (like class, def, if) that serve structural syntax purposes. Attempting to assign a value to a reserved keyword triggers a SyntaxError at parse time before any runtime execution.
  • PEP 8 Conventions:
    • snake_case: Lowercase words separated by underscores (e.g. my_variable). PEP 8 recommends this for variables and function names.
    • UPPER_CASE: Used to convey variables intended to be global constants (e.g., PI_CONSTANT).
    • CamelCase: Capitalized words with no spaces (e.g., MyClass). Standard convention for naming classes.
    • Leading Underscores (e.g., _internal): Signals internal use; controls import behavior (excluded in from module import *).

THEORYRHS Stack Evaluation & The NoneType Singleton

Core Concepts:

Multiple assignment and variable swapping in Python utilize a structured execution order that simplifies logic and reduces errors:

  • RHS First Evaluation: In a multiple assignment statement like a, b = b, a, Python evaluates the entire right-hand side (RHS) before binding any variables on the left-hand side (LHS). At the bytecode level, it pushes the references of b and a to the stack, packs them into a temporary tuple, and then unpacks that tuple into the LHS variables. This prevents value overriding during reassignment.
  • Tuple Unpacking: The comma acts as an implicit tuple builder. a, b, c = 10, 20, 30 assigns matching elements from the tuple to variables by index. A mismatch in element count triggers a ValueError (too many or too few values to unpack).
  • None Singleton: The object None is the sole instance of the class NoneType. It is a singleton—meaning only one copy of None exists in memory. It serves as a placeholder to represent empty states or the default return value of functions that do not have an explicit return statement.

THEORYCPython Integer Caching vs Heap Allocation

Core Concepts:

To optimize execution speed, memory footprint, and garbage collection overhead, CPython implements optimization boundaries for small integers:

  • Pre-allocated Integer Cache: During interpreter startup, CPython initializes an array of integer objects for values in the range [-5, 256]. Any time an integer in this range is declared (e.g., a = 256; b = 256), Python binds the variables to the pre-existing cached objects rather than allocating new memory on the heap.
  • Heap Allocation: Integers outside the cache range (e.g., 257 or -6) are created as separate objects in heap memory. Even if two variables have the same value (e.g., a = 257; b = 257), they point to distinct memory addresses.
  • Identity (is) vs Equality (==):
    • a == b checks value equality (returns True if the numeric values are equivalent).
    • a is b checks identity (returns True only if id(a) == id(b), meaning they occupy the exact same address). Consequently, 256 is 256 yields True, but 257 is 257 yields False.

THEORYType Annotations (PEP 526) & Static Type Checking

Core Concepts:

Type hints provide type declarations for variable declarations, function parameters, and return types, bringing structure to Python's dynamic core:

  • No Runtime Enforcement: Python is inherently dynamic. The interpreter ignores type annotations at runtime. Writing x: int = "hello" runs without raising errors because the annotation is treated as metadata stored inside the module's __annotations__ dictionary.
  • Static Analysis (Mypy / Pyright): Static analysis tools run checks on source code files *before* execution. These tools trace code paths to find type mismatches (e.g., assigning a string to an integer-annotated variable) and highlight errors without needing to run the actual scripts.
  • Code Inspection: You can access runtime types using type(variable) or variable.__class__.__name__. In Python, every class inherits from the root object type, allowing rich runtime introspection.
